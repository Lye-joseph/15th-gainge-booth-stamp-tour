# 부스 스탬프 투어 웹페이지 기능 정의서

## 📋 문서 정보

- **프로젝트명**: 부스 스탬프 투어 (Booth Stamp Tour)
- **버전**: 1.0
- **작성일**: 2025-01-30
- **목적**: 이벤트 참가자가 부스를 방문하여 스탬프를 수집하고 리워드를 신청할 수 있는 모바일 웹 애플리케이션

---

## 1. 프로젝트 개요

### 1.1 프로젝트 목적
- 이벤트 참가자들이 11개의 부스를 방문하며 스탬프를 수집하는 게임형 이벤트 플랫폼
- 스탬프 수집 개수에 따라 선착순 리워드 제공 시스템
- 모바일 기기에서 카메라를 활용한 직관적인 스탬프 수집 경험 제공

### 1.2 주요 특징
- **로컬 기반 스탬프 관리**: 개인정보 수집 없이 브라우저 로컬 스토리지에만 저장
- **실시간 선착순 관리**: Google Sheets 기반 실시간 재고 확인 및 관리
- **QR 코드 연동**: URL 파라미터를 통한 자동 스탬프 수집 지원
- **반응형 디자인**: 다양한 모바일 기기에 최적화된 UI/UX

### 1.3 대상 사용자
- 이벤트 참가자 (주로 모바일 기기 사용)
- 이벤트 운영자 (Google Sheets를 통한 데이터 관리)

6. 사용자 정보 입력
7. 제출 직전 재검증 (GET 요청)
8. 데이터 제출 (POST 요청)
9. Google Sheets에 저장
10. localStorage에 제출 완료 상태 저장

## 3. 주요 기능 상세

### 3.1 스탬프 수집 기능

#### 3.1.1 부스 표시
- **구현 위치**: `index.html` (로드맵 섹션)
- **기능 설명**:
  - 11개의 부스가 세로로 연결된 로드맵 형태로 표시
  - 각 부스는 원형 아이콘과 부스 번호, 회사명 표시
  - 부스 클릭 시 카메라 실행
  - 스탬프가 찍힌 부스는 시각적으로 구분 (체크마크 표시)

#### 3.1.2 카메라 연동
- **구현 위치**: `script.js` (CameraManager 클래스)
- **기능 설명**:
  - HTML5 `<input type="file" accept="image/*" capture="environment">` 활용
  - 모바일 기기에서 카메라 직접 실행
  - 사진 촬영 완료 시 자동으로 스탬프 저장
  - 실제 이미지 파일은 저장하지 않음 (스탬프 상태만 저장)

#### 3.1.3 스탬프 저장
- **구현 위치**: `script.js` (StampStorage 객체)
- **저장 위치**: 브라우저 localStorage
- **저장 형식**:
  ```javascript
  {
    "booth1": true,
    "booth2": false,
    "booth3": true,
    // ...
  }
  ```
- **키 이름**: `boothStamps`
- **특징**:
  - 서버 통신 없이 로컬에만 저장
  - 페이지 새로고침 시에도 상태 유지
  - 개인정보 수집 없음

#### 3.1.4 스탬프 카운터
- **구현 위치**: `index.html` (상단 카운터), `script.js` (updateStampCounter 메서드)
- **기능 설명**:
  - 현재 수집한 스탬프 개수 / 전체 개수(11) 표시
  - 스탬프 수집 시 실시간 업데이트
  - 11개 완주 시 완료 모달 표시

### 3.2 리워드 시스템

#### 3.2.1 리워드 등급
- **티어 11 (치킨 기프티콘)**
  - 조건: 11개 완주
  - 선착순: 5명
  - 리워드 협찬: 콘타 - 코니 캐릭터 신년패키지 (선착순 5명)

- **티어 9 (커피 기프티콘)**
  - 조건: 9개 이상
  - 선착순: 10명
  - 리워드 협찬: 없음

- **티어 7 (에너지 드링크 기프티콘)**
  - 조건: 7개 이상
  - 선착순: 50명
  - 리워드 협찬: 타이거컴퍼니 - 기업 솔루션 3종 세트 (최대 65명)

#### 3.2.2 리워드 신청 자격 확인
- **구현 위치**: `script.js` (checkRewardEligibility 메서드)
- **로직**:
  1. 완료된 스탬프 개수 확인
  2. 7개 미만이면 버튼 비활성화
  3. 이미 제출한 경우 버튼 비활성화 및 텍스트 변경
  4. 자격이 있는 경우 버튼 활성화

#### 3.2.3 선착순 재고 관리
- **구현 위치**: `apps-script.js` (getRemainingCounts 함수)
- **로직**:
  1. Google Sheets에서 모든 제출 데이터 조회
  2. 각 티어별 제출 개수 계산
  3. 한도에서 제출 개수 차감하여 남은 수량 계산
  4. 실시간으로 반환

- **재고 확인 시점**:
  - 모달 열기 전
  - 제출 직전 (경합 상태 방지)

#### 3.2.4 티어 자동 할당
- **구현 위치**: `script.js` (openRewardModal 메서드)
- **로직**:
  1. 사용자의 완료 개수로 자격 있는 최고 티어 확인
  2. 재고 확인
  3. 자격 있는 티어부터 순차적으로 재고 확인
  4. 재고가 있는 가장 높은 티어 할당
  5. 자격은 있으나 재고가 없으면 하위 티어로 안내

#### 3.2.5 중복 제출 방지
- **구현 위치**: 
  - 프론트엔드: `script.js` (RewardStorage 객체)
  - 백엔드: `apps-script.js` (checkDuplicateSubmission 함수)
- **검증 기준**:
  - 이메일 주소 (대소문자 무시, 공백 제거)
  - 전화번호 (하이픈, 공백 제거)
- **검증 시점**:
  - 프론트엔드: localStorage 확인 (즉시)
  - 백엔드: Google Sheets 확인 (서버 측 최종 검증)

### 3.3 리워드 신청 폼

#### 3.3.1 입력 필드
- **필수 필드**:
  - 이름 (userName)
  - 회사명 (companyName)
  - 연락처 (phoneNumber) - 형식: 010-1234-5678
  - 이메일 (email) - 이메일 형식 검증

- **선택 필드**:
  - 직급 (position) - 예: 매니저 / 팀장 / 주임

#### 3.3.2 폼 검증
- **클라이언트 측 검증**:
  - HTML5 required 속성 활용
  - 이메일 형식 검증 (type="email")
  - 전화번호 형식 검증 (type="tel")

- **서버 측 검증**:
  - 필수 필드 누락 확인
  - 중복 제출 확인
  - 재고 확인

#### 3.3.3 제출 프로세스
1. 폼 제출 버튼 클릭
2. 버튼 비활성화 및 "등록 중..." 표시
3. 제출 중 플래그 설정 (중복 제출 방지)
4. 재고 재검증
5. 데이터 제출 (POST 요청)
6. 성공 시:
   - localStorage에 제출 완료 상태 저장
   - 버튼 비활성화 및 텍스트 변경
   - 성공 메시지 표시
   - 모달 닫기
7. 실패 시:
   - 에러 메시지 표시
   - 버튼 다시 활성화

### 3.4 URL 파라미터 기능

#### 3.4.1 자동 스탬프
- **형식**: `?booth=3`
- **기능**: 해당 부스의 스탬프를 자동으로 찍음
- **사용 사례**: QR 코드를 통한 자동 체크인
- **구현 위치**: `script.js` (attachPathBasedStamp 메서드)

#### 3.4.2 개발 모드
- **형식**: `?dev=1`
- **기능**: 테스트 초기화 버튼 표시
- **사용 사례**: 개발 및 테스트 시 데이터 초기화

#### 3.4.3 초기화 모드
- **형식**: `?reset=1`
- **기능**: 모든 스탬프 및 리워드 제출 상태 초기화
- **사용 사례**: 테스트 시 전체 초기화

## 4. 사용자 시나리오

### 4.1 일반 사용자 시나리오

#### 시나리오 1: 스탬프 수집
1. 사용자가 웹페이지 접속
2. 부스 1 클릭
3. 카메라 실행
4. 사진 촬영
5. 스탬프 자동 저장 및 UI 업데이트
6. 카운터: 1/11 완료
7. 부스 2~11까지 반복
8. 11개 완주 시 완료 모달 표시

#### 시나리오 2: 리워드 신청 (성공)
1. 사용자가 7개 이상 스탬프 수집
2. "상품수령 정보 등록하기" 버튼 활성화
3. 버튼 클릭
4. 모달 열림, 사용 가능한 티어 표시
5. 정보 입력 (이름, 회사명, 연락처, 이메일, 직급)
6. "정보 등록하기" 버튼 클릭
7. 재고 확인 및 제출
8. 성공 메시지 표시
9. 버튼 텍스트 변경: "이미 상품수령 정보를 등록하셨습니다"

#### 시나리오 3: 리워드 신청 (재고 부족)
1. 사용자가 11개 완주
2. 리워드 신청 버튼 클릭
3. 모달 열림
4. 재고 확인 결과 11개 완주 상품 마감
5. 알림: "11개 완주자 상품 수령 선착순 등록이 마감되었습니다. 9개 이상 상품 수령 등록으로 안내드립니다."
6. 9개 이상 티어로 자동 할당
7. 정보 입력 및 제출

#### 시나리오 4: 중복 제출 시도
1. 사용자가 이미 리워드 신청 완료
2. 버튼 클릭 시도
3. 버튼 비활성화 상태이므로 클릭 불가
4. 또는 모달 열기 시도 시 알림 표시

### 4.2 QR 코드를 통한 체크인
1. 부스에 QR 코드 부착
2. 사용자가 QR 코드 스캔
3. URL: `https://example.com/?booth=3`
4. 자동으로 부스 3 스탬프 찍힘
5. 알림: "부스 3 스탬프가 자동으로 찍혔습니다! 🎉"

## 5. 데이터 구조

### 5.1 localStorage 구조

#### 스탬프 데이터
```javascript
// 키: "boothStamps"
{
  "booth1": true,
  "booth2": true,
  "booth3": false,
  // ...
  "booth11": false
}
```

#### 리워드 제출 상태
```javascript
// 키: "rewardSubmitted"
"true" // 또는 없음 (제출 안 함)
```

### 5.2 Google Sheets 데이터 구조

#### 시트 이름
- `리워드신청`

#### 컬럼 구조
| 컬럼명 | 데이터 타입 | 설명 | 필수 여부 |
|--------|------------|------|----------|
| 이름 | 문자열 | 참가자 이름 | 필수 |
| 직급 | 문자열 | 참가자 직급 | 선택 |
| 회사명 | 문자열 | 참가자 회사명 | 필수 |
| 연락처 | 문자열 | 전화번호 (010-1234-5678) | 필수 |
| 이메일 | 문자열 | 이메일 주소 | 필수 |
| 완료개수 | 숫자 | 수집한 스탬프 개수 (7~11) | 필수 |
| 리워드등급 | 문자열 | "치킨" / "커피" / "에너지드링크" | 필수 |
| 제출시간 | 문자열 | ISO 8601 형식 (2025-01-30T09:00:00.000Z) | 자동 |

### 5.3 API 요청/응답 구조

#### GET 요청 - 전체 제출 목록 조회
```
GET {WEB_APP_URL}?action=list
```

**응답**:
```json
{
  "ok": true,
  "rows": [
    {
      "name": "홍길동",
      "position": "매니저",
      "company": "가인지컨설팅그룹",
      "phone": "010-1234-5678",
      "email": "hong@example.com",
      "completedCount": 11,
      "rewardLevel": "치킨",
      "timestamp": "2025-01-30T09:00:00.000Z"
    }
  ]
}
```

#### GET 요청 - 남은 수량 조회
```
GET {WEB_APP_URL}?action=remaining
```

**응답**:
```json
{
  "ok": true,
  "remaining": {
    "tier11": 3,
    "tier9": 8,
    "tier7": 45
  }
}
```

#### GET 요청 - 특정 티어 확인
```
GET {WEB_APP_URL}?action=check&tier=tier11
```

**응답**:
```json
{
  "ok": true,
  "available": true,
  "remaining": 3
}
```

#### POST 요청 - 리워드 신청 제출
```
POST {WEB_APP_URL}
Content-Type: application/json
```

**요청 본문**:
```json
{
  "name": "홍길동",
  "position": "매니저",
  "company": "가인지컨설팅그룹",
  "phone": "010-1234-5678",
  "email": "hong@example.com",
  "completedCount": 11,
  "rewardLevel": "치킨",
  "timestamp": "2025-01-30T09:00:00.000Z"
}
```

**응답 (성공)**:
```json
{
  "ok": true,
  "message": "리워드 신청이 완료되었습니다.",
  "remaining": {
    "tier11": 2,
    "tier9": 8,
    "tier7": 45
  }
}
```

**응답 (실패 - 중복 제출)**:

## 8. 보안 및 개인정보 보호

### 8.1 개인정보 수집 최소화
- **스탬프 데이터**: 로컬에만 저장, 서버 전송 없음
- **리워드 신청 데이터**:

  - 리워드 전달 목적으로만 사용
  - Google Sheets에 저장 (Google의 보안 정책 적용)

### 8.2 데이터 보호
- **로컬 데이터**: 브라우저 localStorage (사용자 기기 내)
- **서버 데이터**: Google Sheets (Google 계정 보안)
- **전송 암호화**: HTTPS 통신

### 8.3 중복 제출 방지
- **클라이언트 측**: localStorage 기반 검증
- **서버 측**: 이메일/전화번호 기반 중복 검증
- **제출 중 플래그**: 동시 제출 방지

### 8.4 재고 경합 방지
- **제출 직전 재검증**: 최종 재고 확인
- **원자적 연산**: Google Sheets의 트랜잭션 특성 활용

## 9. 오류 처리

### 9.1 네트워크 오류
- **재고 조회 실패**: 
  - Fallback: 전체 목록 조회 후 로컬 계산
  - 실패 시 기본값 사용 (전체 한도로 가정)
- **제출 실패**:
  - 에러 메시지 표시
  - 버튼 다시 활성화
  - 로컬 백업 저장 (선택적)

### 9.2 데이터 검증 오류
- **필수 필드 누락**: 클라이언트 측 HTML5 검증
- **이메일 형식 오류**: HTML5 이메일 타입 검증
- **중복 제출**: 서버 측 검증 및 에러 메시지

### 9.3 브라우저 호환성
- **localStorage 미지원**: 콘솔 에러 로그 (기능 제한)
- **카메라 미지원**: 파일 선택 대화상자로 대체
